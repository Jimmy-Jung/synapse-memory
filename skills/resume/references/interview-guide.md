# Cold-start 인터뷰 가이드 (interview-guide)

`/sm:resume` Phase 0 의 **첫 사용 분기**용. 사용자가 처음 이력서를 만들고 기존 이력서·자소서·`Entities/Projects/*` 카드가 없을 때, 구조화 인터뷰로 경력 사실을 수집한다.

목표: `resume-template.md` 품질을 채울 수 있을 만큼의 1차 자료를 모으는 것. 결과는 `Companies/<slug>/interview-notes.md` 에 저장하고(또는 Profile 갱신) Phase 1~4 의 입력으로 쓴다.

## 진행 원칙

- **한 번에 한 묶음씩** 묻는다. 답을 받으면 다음 묶음. 6묶음 전부를 한꺼번에 쏟지 않는다.
- **프로젝트(블록 C)를 가장 깊게** 판다. 여기서 이력서 품질이 갈린다. 막연한 답엔 "구체적으로 무엇을, 왜 그렇게?"를 되묻는다.
- **수치는 측정 맥락까지** 캐묻는다("그 90%는 어떻게 쟀나요? 운영 표본인가요, 내부 시연인가요?"). 모르면 정성적으로 남기고 추측하지 않는다(style-guide 정직성).
- AskUserQuestion 또는 자유 대화 중 상황에 맞는 방식을 쓴다. 답이 비면 비운 채로 둔다.

## 질문 묶음

### A. 기본 정보
- 이름 / 지원 직무 / 총 경력 연차 / 연락처·이메일
- 블로그·GitHub·포트폴리오 링크

### B. 회사별 경력 (최근 → 과거)
회사마다: 회사명 / 근무 기간 / 고용 형태 / 직무·직급 / 팀 규모 / 회사가 하는 일 / 내가 담당한 범위(앱·서비스)

### C. 프로젝트 심층 (회사당 2~4개, **STAR**)
프로젝트마다:
- **문제**: 무엇이 왜 문제였나? (비즈니스/기술 맥락)
- **역할**: 내가 한 구체 행동과 설계 결정, 그렇게 한 이유
- **해결**: 가장 까다로웠던 지점과 푼 방법
- **성과**: 결과 — 수치가 있으면 before→after + 측정 맥락. GitHub/공개 자료 있으면 링크.

### D. 핵심 역량·강점
- 여러 프로젝트에서 반복되는 강점 패턴(예: 레거시 개선, 모듈화, 운영 자동화)
- 새 기술 도입 시 본인의 판단 기준

### E. 학력·자격·기타
- 학력(학교/전공/기간/상태)
- 자격증·교육 수료·수상·특허
- 오픈소스/기술 발표/스터디 운영/개발 블로그 등

### F. 지향·관심 (간단한 소개용)
- 왜 이 일을 시작했고, 어떤 방식으로 일하는 것을 선호하는가
- 최근 관심사·학습 방향

## 마무리

- 수집 결과를 `interview-notes.md` 로 정리(블록 A~F 구조 유지, 출처/링크 보존).
- 사용자에게 빠진 부분·불확실한 수치를 확인받는다.
- 이후 Phase 1(회사 조사) → Phase 3 초안에서 이 노트를 1차 자료로 사용. 인터뷰로 드러난 프로젝트는 추후 `Entities/Projects/` 카드로 승격할 수 있음을 안내(자동 생성하지 않음).
