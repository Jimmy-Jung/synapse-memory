# 이력서 스타일 가이드 (style-guide)

`resume-template.md` 를 채울 때의 톤·문장·정량지표 규칙. 기준 샘플 `정준영 2026_개선 0514.md` 에서 역추출했다.

## 톤

- 차분하고 근거 중심. "화려함" 대신 "변경의 부수 효과를 얼마나 줄였는가, 후속 개발자가 이어갈 수 있는가".
- 1인칭 존댓말("~했습니다", "~합니다"). 과장·형용사 남발 금지.
- 자랑이 아니라 의사결정의 서술. "무엇을 왜 그렇게 했는가"가 드러나야 한다.

## 프로젝트 STAR 4블록 (고정)

각 주요 프로젝트는 **문제 → 역할 → 해결 → 성과** 순서.
- **문제**: 비즈니스/기술 맥락. "왜 그게 문제였나". (예: 외부 SDK가 실기기 바이너리만 제공 → 시뮬레이터 빌드 실패)
- **역할**: 내가 한 구체 행동 + 설계 이유. 동사로 시작. 3~4개.
- **해결**: 가장 까다로웠던 지점 1개를 골라 어떻게 풀었는지 서사로.
- **성과**: 결과. 가능하면 before→after 수치. 추측 불가.

## 정량지표 정직성 (위반 금지)

- 카드·자료에 **실재하는 수치만** 사용. 없으면 정성적으로 쓰고 지어내지 않는다.
- 측정 맥락을 함께 적는다. 샘플에서 검증된 표현:
  - "내부 시연 환경 측정 시 ... 약 90% (개인 측정·운영 표본 미수집)"
  - "같은 유형의 버그 평균 수정 시간 2시간 → 35분 수준으로 단축 (팀 내 약 20건 샘플 기준)"
  - "유닛 테스트 커버리지 0% → 32% (ViewModel·UseCase 레이어 기준)"
- 운영 표본이 없으면 "개인 측정", "내부 시연 기준"을 명시. 회사 공식 지표인 척 금지.

## 배치 규칙

- 지원 회사 키워드와 매칭되는 프로젝트/역량을 **상단**에 배치.
- 한 줄 소개·핵심 역량에 회사 도메인 키워드를 자연스럽게 1~2개 녹인다(억지 금지).
- 회사가 직접 경험이 없는 도메인이면(예: O2O), 인접 역량(고트래픽 안정성·모듈화·협업)을 전면에.

## 근거·출처

- 모든 프로젝트 주장은 `Entities/Projects/*` 카드에 근거가 있어야 한다.
- 자료에 없는 사실은 쓰지 않는다. 누락하거나 "정보 없음"으로 둔다.
- 초안(resume-draft.md)에서는 각 항목에 출처 프로젝트를 `[[Entities/Projects/X]]` 로 링크. 최종(resume.md)에서는 자연스러운 문장으로 녹이되 사실은 보존.

## 길이 감각 (샘플 기준)

- 한 줄 소개 1문장 / 경력 요약 2~4문장 / 핵심 역량 4~6 bullet / 간단한 소개 2~3문단.
- 회사당 주요 프로젝트 2~4개. 프로젝트당 역할 3~4 bullet, 성과 3~5 bullet.
- 전체 분량은 샘플(약 330줄)에 준한다. 너무 짧으면 깊이 부족, 너무 길면 산만.
